# 2) Data Model (Prisma + RLS)
Create Prisma models with tenant-aware RLS:
- duty_users (per-tenant app users table if not already mapped via memberships)
  id, tenant_id, username, role: 'admin'|'member', active, profile_json, created_at
  # link to platform users/memberships if present; otherwise map via user_id
- duties
  id, tenant_id, title, desc, schedule ENUM('onetime','daily_not_sun','daily_not_sat_sun','weekly_before_sat','monthly_before_5','monthly_before_28'),
  priority ENUM('high','medium','low'), status ENUM('pending','for_approval','completed'),
  assignees string[] (usernames or user_ids), created_by, need_approval boolean,
  due_at timestamptz, created_at timestamptz, last_action, last_action_at timestamptz, last_action_note text,
  series_id text, occurs_on date
- holidays
  id, tenant_id, date, label, is_optional boolean, created_by
- approvals
  id, tenant_id, duty_id, requested_by, decided_by, decision ENUM('approved','rejected','none'), decided_at, note
- backups (optional server-side)
  id, tenant_id, snapshot_json, created_at, created_by
RLS: only members of a tenant can CRUD their tenant’s rows; editors/admins enforce per-action rules.

  - (if needed) org-scoped login/impersonate endpoints (follow platform conventions)
/api/wytDuty/duties

  - GET    list duties (filters: status, priority, assignee, date range)

  - PATCH  :id (update fields, state machine validation)

  - POST   :dutyId/approve
  - POST   :dutyId/reject

/api/wytDuty/reports

# 6) Scheduling & Recurrence

- monthly_before_5 / monthly_before_28 → compute next month’s cut-off
Store series_id to tie instances. Background job/cron stub: on user interaction or periodic task, materialize next occurrences.
